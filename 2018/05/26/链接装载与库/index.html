第1章 温故知新

看介绍作者也是理工男，所以思维简单粗暴，直接从问题引出本书的内容。
针对hello world问题的追问？私以为这些问题可以用来区分一个编程人员的技术敏感度

  • 程序为什么要被编译器编译之后才可以运行？
  • 编译器在把C语言转换成可以执行的机器码的过程中做了什么？怎么做的?
  • 最后编译出来的可执行文件里面是什么？除了机器码还有什么？它们是怎么存放的？怎么组织的？
  • #include<stdio.h>是什么意思？把stdio.h包含进来意味着什么？C语言库又是什么？它是怎么实现的？
  • 不同的编译器和不同的硬件平台，以及不同的操作系统，最终编译出来的结果一样吗？为什么？
  • hello world程序是怎么运行起来的？操作系统怎么装载它的？它从哪里开始执行?到哪结束？main函数之前发生了什么？main函数结束之后又发生了什么？
  • 如果没有操作系统，hello world可以运行吗？如果要在一台没有操作系统的机器上运行hello world需要什么？应该怎么实现？
  • printf是怎么实现的？他为什么可以有不定数量的参数？为什么它能够在终端上输出字符串？
  • hello world程序在运行时，在内存中是什么样子的？

线程

轻量级进程。
IO密集型线程、CPU密集型线程。

线程安全

竞争和原子操作：单指令的操作为原子的，因为无论如何，单条指令的执行是不会被打断的。
同步与锁：信号量、互斥量、临界区、读写锁、条件变量。
可重入与线程安全
可重入的情况：多个线程同时执行一个函数或者函数自身调用自身。
可重入函数必须具备的特点：

  • 不使用任何（局部）静态或者全局的非const变量。
  • 不返回任何（局部）静态或全局的非const变量的指针。
  • 仅依赖与调用方提供的参数。
  • 不依赖任何单个资源的锁
  • 不调用任何不可重入的函数。

过度优化：注意即使编程时合理使用了锁，也不一定能够保证线程安全。原因是编译器的过度优化和CPU的动态调度换序。
使用volatile关键字可以1)阻止编译器为了提供速度将一个变量缓存到寄存器内而不写回。2）阻止编译器调整操作volatile变量的指令顺序。
可以解决编译器的过度优化问题，但是无法阻止CPU的动态调用换序。
阻止CPU动态调度换序的方案是使用CPU提供barrier指令，不同的CPU名称各不相同。

多线程模型

一对一线程模型、多对一模型、多对多模型（多映射到少）。

被隐藏的过程

现在的集成环境屏蔽了很多处理过程。这些处理过程的理解对于解决开发过程中遇到的问题尤其重要。
预处理（prepressing）、编译(compilation)、汇编（assembly）、链接（linking）
预编译主要处理源文件中以#开始的预编译指令。例如#include #define等。

  • 将原来的#define 删除，并且展开所有的宏定义。
  • 处理所有的条件预编译指令。#if #ifdef #elif #else #endif等等等。
  • 处理#include，将被包含的文件插入到该预编译指令的位置。
  • 删除所有的注释
  • 添加行号和文件名表示。便于编译时产生调试用的行号信息。
  • 保留所有的#pragma编译器指令，因为后面编译器需要使用。

第3章 目标文件里有什么

目标文件的格式

window下为PE linux下为ELF。
ELF文件的格式有：

  1. 可重定位文件。.o
  2. 可执行文件。
  3. 共享目标文件。.so
  4. 核心转储文件。coredump

tips：可以使用file命令查看对应的文件格式。

目标文件是什么样的

目标文件以“段”的形式组织。比如机器指令存放在代码段（.text或.code）全局变量和局部静态变量存放在数据段（.data）
具体的组织格式如下：

  1. 文件头，描述整个文件的文件属性。包括是否可执行。静态还是动态链接以及入口地址、目标硬件、目标操作系统等。还包括一个段表。描述了各个段在文件中的偏移位置及段的属性。
  2. 各个段的数据。代码段、数据段、bss段等等。

tips:可以使用objdump -h命令来分析.o 文件
tips：可以使用size命令查看elf文件的代码段、数据段、BSS段的长度

代码段

tips：可以使用objdump -s 命令将所有段的内容以十六进制形式打印出来。 -d 参数可以将所有包含指令的段反汇编。
代码段存放的是机器指令。通过 objdump -s -d 命令可以很自然的看出来。

数据段和只读数据段

.data保存的是已经初始化了的全局静态变量和局部静态变量。
.rodata 保存一般是程序里面的只读变量和字符串变量。

BSS段

存放的是未初始化的全局变量和局部静态变量。

其他常见段

.comment 存放编译器版本信息。
.debug 存放调试信息。
.dynamic 存放动态链接信息。
.line 存放调试时的行号表。
.shstrtab section string table 段名表。

ELF文件结构描述

总体结构从前向后分别为

  1. ELF 文件头。描述了整个文件的基本属性。
  2. .text
  3. .data
  4. .bss
  5. other sections
  6. Section header table段表。

段表

段表是除了ELF头部以外最重要的结构，它描述的是ELF的各个段的信息。比如每个段的段名、长度、在文件中的偏移、读写权限以及段的其他属性。

重定位表

在上面例子中有个叫.rela.text的段。是一个重定位表。链接器在处理目标文件时，需要对目标文件中代码段和数据段中对绝对地址的引用的位置进行重新定位。这些需要重定位的信息就保存在重定位表里面。

字符串表

ELF文件中用到了很多字符串，比如段名、变量名、函数名。因为字符串的长度是不固定的，所以用固定的结构表示比较困难。常见的做法是将字符串集中起来存放。然后用字符串在表中的偏移来引用字符串。
常见的段为.strtab和.shstrtab。分别为字符串表和段表字符串表。

链接的接口—符号

目标文件中的符号大体分成以下几类

  • 定义在本目标文件中的全局符号，可以被其他目标文件引用。
  • 在本目标文件中引用的全局符号，却没有定义在本目标文件中。
  • 段名
  • 局部符号，只在编译单元内部可见。
  • 行号信息。

链接过程关注的只有前面两类符号，也就是全局符号。

符号表结构体

查看符号表的方法有下面三种。常用readelf -s
符号表段名一般叫.symtab 结构体为Elf32_Sym
常见的字段有

  • 符号类型和绑定信息
  • 符号所在的段
  • 符号值

特殊符号

链接器在链接生成可执行文件时，会定义很多特殊符号。这些特殊符号也可以在代码中进行声明和引用。
常见的特殊符号有

  • __executable_start 程序起始地址。
  • __etext/_etext/etext 代码段结束地址
  • _edata/edata 数据段结束地址
  • _end/end 程序结束地址

符号修饰和函数签名

为了防止代码中的符号名冲突，Unix下的C语言规定，经过编译后的符号名前面加下划线。
后来C++语言在设计的时候就考虑到这个问题，所以增加了名称空间（namespace）的方法。
现在Linux下C语言貌似已经去掉了加下划线这种方式。windows平台的C语言还是保持下划线这种传统。
C++符号修饰
为了支持C++中的重载等复杂特性。增加了符号修饰或者叫符号改编的机制。
编译后函数会依据一定的规则生成修饰后的函数名称。

extern “C”

由于C++的符号修饰机制C语言不支持，所以为了兼容C语言，会通过extern “C”关键字来通知编译器，将带有这个标志的代码当成C语言处理，符号修饰机制将不会起作用。

弱符号和强符号

C/C++中，编译器默认函数和初始化的全局变量为强符号、未初始化的全局符号为弱符号。可以通过关键字”attribute((weak))”来定义。
针对强弱符号的概念。链接器的处理规则为：

  • 不允许强符号被多次定义。
  • 如果一个符号在目标文件中是强符号，在其他文件中是弱符号，那么选择强符号。
  • 如果一个符号在所有文件中都是弱符号，那么选择其中占用空间最大的那个。

弱引用和强引用

对外部目标文件的符号引用在目标文件被编译成可执行文件时，必须要被正确决议。如果没有找到该符号的定义。链接器就会报符号未定义错误。
这种被称为强引用。
反之，还有一种弱引用。在处理时，如果该符号有定义，则引用，如果未定义则对于这个符号的引用不报错。
可以使用关键字“attribude((weakref))”来对一个外部函数声明为弱引用。
这种弱符号和和弱引用 对于库来说非常有用。比如库中定义的弱符号可以被用户定义的强符号覆盖。从而使程序可以使用自定义版本的库函数。或者程序可以对某些扩展功能的引用定义为弱引用。当将扩展模块与程序链接到一起的时候，功能模块就可以正常使用。当去掉某些功能模块时，程序也可以正常链接，只是缺少了相应的功能。这使得功能更加容易采集和组合。

调试信息

gcc编译时，加上-g参数，编译器就会在目标文件中加上调试信息。这些调试信息都是以段的形式存放的。现在的ELF文件采用一种叫DWARF的标准调试信息格式。
因为调试信息占用相当大的空间，所以当开发完成需要将程序发布的时候，linux下可以使用strip命令去掉ELF文件中的调试信息。
